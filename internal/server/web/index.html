DPI bypass

Fragment the real ClientHello (splits the SNI across TCP writes — the reliable userspace bypass) and optionally inject fake ClientHellos with a corrupted segment so a DPI desyncs. Fake injection needs Linux root or Windows+WinDivert; without it, fragmentation still applies.

wrong_checksum / wrong_seq craft raw packets and require elevated privileges (Linux root, or WinDivert on Windows). When unavailable the proxy logs it and falls back to fragmentation only.

Google Tunnel (Fastly) NEW

A domain-fronting tunnel: your traffic leaves your machine looking like an ordinary TLS connection to www.google.com, rides through a Google Apps Script relay you deploy, then a Cloudflare Worker fetches the real site. The network only ever sees Google. Everything below is generated by V2RayEz — you never touch any external repo.

How it works

Browser→ V2RayEz proxy→ TLS · www.google.com→ Apps Script→ CF Worker→ Target

① Generate your scripts

Deploy worker.js to a Cloudflare Worker, and Code.gs to a Google Apps Script Web App (Execute as: Me · Access: Anyone). Copy the Apps Script Deployment ID into step ②.

② Connect

The front must sit on the same CDN edge as your relay host for fronting to work. The Google presets all share Google's edge with Apps Script, so they work out of the box — pick any to vary what DPI sees. Fastly is experimental and only works if your relay is Fastly-hosted; otherwise keep a Google preset.

③ Point your apps at it

  1. Set your browser/system HTTP proxy (HTTP and HTTPS) to: 127.0.0.1 : 8086
  2. For HTTPS sites, install & trust the V2RayEz Google-Tunnel CA once (the tunnel decrypts locally to relay each request).
The CA lets the tunnel read your HTTPS requests on this device only, to relay them — it never leaves your machine. Streaming/WebSocket isn't supported by the relay. Deploy and use within Google's and Cloudflare's terms and your local laws.

Domain Fronting NEW

Fully client-side domain fronting — no server, no relay. A local proxy decrypts your own HTTPS using a CA you install once, reads the real Host, then reconnects to that site's CDN edge showing an allowed front domain (e.g. www.google.com) to the network while carrying the real Host inside. The censor only sees ordinary traffic to the front.

How it works

Browser→ V2RayEz MITM→ TLS · www.google.com→ CDN edge (real Host inside)→ Real site

① Front rules (host → front domain)

One rule per line: host-suffix = front-SNI, or host-suffix = front-SNI = dial-host to also redirect the TCP connection (needed for Fastly). With no dial-host, it connects to the real site's IP and only swaps the TLS SNI. Edit freely; leave it as-is for the built-in set.

Real hosts are resolved over fronted DoH so the censor's poisoned DNS (e.g. a 10.10.34.x blackhole) is never used. Defaults front Cloudflare 1.1.1.1; change only if needed.

SPlus Tunnel NEW

Tunnels TCP through a SoroushPlus voice-call LiveKit data channel (port of theermia/SPlusTunnel). The server runs on free internet; the client runs restricted and exposes a local SOCKS5 proxy. Both join the same call.

How to get the token

  1. Start a voice call inside SoroushPlus Messenger (web client) between the two devices.
  2. Open DevTools (F12) → Network → filter WS, find the connection to k.splus.ir:8446.
  3. Copy the access_token query parameter from that WebSocket URL — that is the token.
  4. Start the server side first, then the client. Each side needs its own token.

Using the SOCKS5 proxy (client)

Point apps at socks5h://<host>:<port> and route DNS through the proxy. Quick test:

Under the hood

A single self-contained Go binary serves this control panel on loopback. No external assets are loaded — everything runs fully offline, which matters for a circumvention tool. The live console below streams status from every action over Server-Sent Events.

Scope & safety

The proxy intentionally skips upstream certificate verification — that is inherent to SNI spoofing, where the presented certificate won't match the fake SNI. Use this tool only where it is lawful for you to do so.

The SPlus tunnel needs a LiveKit-enabled build (go build -tags livekit). Without it, starting a tunnel returns build instructions.
